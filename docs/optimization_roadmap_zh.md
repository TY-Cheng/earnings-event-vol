# 优化路线图（中文）

## 我对这个项目下一阶段优化重点的判断

我会把这个项目下一阶段的优化重点，按价值排序放在下面这 5 个方向上：

| 优先级 | 改进方向 | 我为什么把它排在这里 |
| --- | --- | --- |
| 1 | `execution confidence score` | 先确认收益不是来自不可成交价格，否则后面所有 alpha 讨论都站不住。 |
| 2 | `event regime segmentation` | 财报事件不是同一种机制，把不同 regime 混成平均值，很容易把真实结构稀释成噪音。 |
| 3 | `model vs IVAR defeat analysis` | 这个项目真正要回答的不是“模型准不准”，而是“模型在哪些条件下真的打败了市场共识”。 |
| 4 | `edge confidence calibration` | 从“模型给出一个 edge”升级成“模型知道这个 edge 有多可靠”。 |
| 5 | `false positive / false negative casebook` | 这是最值钱的研究资产，因为失败案例会暴露模型到底是在理解结构，还是在拟合幻觉。 |

如果只讲一句话，我的核心观点是：

> 这个项目下一步不应该优先继续堆复杂模型，而应该优先提高执行可信度、结构分层能力和失败样本沉淀能力。

---

## 为什么这样排序

这个仓库最有价值的地方，本来就不是“做一个更复杂的波动率模型”，而是把下面这条链路做完整：

```text
财报事件定义
-> 市场隐含事件方差 IVAR_event
-> realized event variance
-> premium-space edge
-> 策略判断与代理 PnL
```

所以我判断优化优先级时，标准不是“哪个模型更炫”，而是下面三个问题：

1. 这个改进会不会提高研究结论的可信度？
2. 这个改进会不会让我更清楚地知道模型为什么有效、为什么失效？
3. 这个改进会不会沉淀成下一轮研究可以复用的资产？

按照这个标准，执行可信度、regime 分层、IVAR defeat 分析、置信度校准和失败案例库，都会比继续加新模型更值钱。

---

## 1. Execution Confidence Score

### 我为什么把它排第一

我最先想解决的问题，不是模型误差，而是价格真实性。

这个仓库当前的研究价值很大程度建立在 `trade_proxy` 上。如果代理入场价和代理出场价对应的是“看起来存在、但实际上很难成交”的价格，那么后面所有 edge、PnL、排序优势都可能只是价格幻觉。

换句话说：

> 只要 execution layer 不够可信，后面所有 alpha 讨论都要打折。

### 我想解决的核心问题

- 这笔代理交易对应的价格到底多可成交？
- 这个收益是不是来自流动性空洞、开盘噪音或者极端窗口选择？
- 哪些样本应该被保留为高可信信号，哪些只能作为低可信研究样本？

### 我建议怎么落地

我会给每个事件样本增加一个 `execution_confidence_score`，作为交易代理可信度标签。这个分数可以由几类信息组成：

- 入场窗口成交量是否足够
- 价格是否来自单点成交还是稳定 VWAP
- 开盘后锚定窗口是否过于稀疏
- 收盘前退出窗口是否有足够连续观测
- 同合约日频 close 与秒级窗口 VWAP 是否严重背离
- 价差代理、spread-over-mid 或局部 IV 是否异常
- 是否触发了 fallback 逻辑

### 我希望最终看到什么

- 每个事件有一列 `execution_confidence_score`
- 评估报告默认分层展示高/中/低执行可信度结果
- 主要策略指标优先看高可信子样本
- 低可信样本不直接删除，但要显式标注

我认为这是把“研究 proxy”往“更可相信的交易 proxy”推进的第一步。

---

## 2. Event Regime Segmentation

### 我为什么把它排第二

财报事件不是同质样本。

盘前和盘后不是一回事，高 IV 和低 IV 不是一回事，大市值和小市值不是一回事，宏观高波动阶段和低波动阶段更不是一回事。如果把这些机制完全混在一起看一个平均结论，最后往往会出现两种问题：

- 真正有用的结构被平均掉
- 本来只在局部成立的规律被误判成普遍规律

### 我想解决的核心问题

- 模型到底在哪些 regime 下有效？
- 哪些事件机制本身就不应该被同一个模型同权看待？
- 平均 alpha 背后是不是掩盖了严重的子样本异质性？

### 我建议怎么落地

我会优先做一套轻量但高价值的 regime 切分：

- 公告时点：`BMO` / `AMC`
- 市场环境：高 VIX / 低 VIX
- 事件定价状态：高 `IVAR_event` / 低 `IVAR_event`
- 股票特征：大市值 / 中小市值
- 流动性状态：高成交活跃 / 低成交活跃
- 跳空主导 vs 盘中消化主导

### 我希望最终看到什么

- 指标表不只看整体，还看主要 regime 切片
- 失败案例库能自动带上 regime 标签
- 后续特征和模型可以围绕 regime 做有针对性的增强

我不把这一步理解成“多做几张分组表”，而是把它当成结构识别工具。

---

## 3. Model vs IVAR Defeat Analysis

### 我为什么把它排第三

这个项目和很多普通预测项目不一样，它从一开始就有一个非常明确的对手：

> 市场自己给出的 `IVAR_event`

所以我最关心的，不是模型绝对误差是不是更小，而是下面这个问题：

> 模型到底在什么条件下，稳定地打败了市场的事件定价？

如果没有这一层分析，我们只能说“模型好像有点用”，但没法说清楚它到底赢在哪里。

### 我想解决的核心问题

- 模型在哪些事件上显著优于 `IVAR_event`？
- 模型在哪些事件上只是重复了市场已经知道的东西？
- 模型的增量价值来自哪里，是历史事件、SEC 基本面、序列信息，还是某类 regime 切片？

### 我建议怎么落地

我会补一层专门的 `defeat analysis`：

- 比较模型预测与 `IVAR_event` 的相对增量价值
- 分层看模型在哪些事件中改变了排序
- 看“模型纠正市场”的样本和“模型被市场纠正”的样本
- 拆解增量收益是否集中在特定事件类型

### 我希望最终看到什么

- 一组面向研究问题本身的结果表
- 明确知道“模型是怎么赢的”
- 明确知道“模型什么时候根本没赢”

我觉得这一步会让项目的研究表达更像“在挑战市场定价”，而不是“在做又一个预测模型”。

---

## 4. Edge Confidence Calibration

### 我为什么把它排第四

现在的系统已经能输出 edge，但 edge 本身还不等于可靠度。

两个样本都可能给出相同大小的 edge，可其中一个 edge 背后有很稳定的历史结构支持，另一个只是偶然噪音。只输出一个点预测，会让系统在决策层面仍然比较粗糙。

### 我想解决的核心问题

- 模型给出的 edge 有多可靠？
- 哪些信号适合更大权重，哪些只适合观察？
- 预测强度和预测可信度能不能被拆开？

### 我建议怎么落地

我会给 edge 再加一层 calibration：

- 预测残差分桶
- 不同 edge 区间的 realized hit rate
- 不同 regime 下 edge 的稳定性
- 高 edge 但低稳定性的样本标记
- 输出 `edge_confidence_band` 或类似字段

### 我希望最终看到什么

- 模型不是只告诉我“有信号”
- 模型还能告诉我“这个信号多值得信”
- 策略构建时可以按 confidence 而不是只按 edge 排序

这一步是把研究信号升级成可分级决策对象。

---

## 5. False Positive / False Negative Casebook

### 我为什么认为它最值钱

很多研究系统只保存成功回测。

但我更看重失败样本，因为失败样本才会告诉我：

- 模型到底是在理解结构，还是在拟合幻觉
- 市场到底在哪些地方比模型更聪明
- 哪类信号是“看起来强、实际上脆弱”
- 哪类 regime 或样本标签应该进入下一轮过滤器和特征工程

成功样本只能说明“系统有时能赢”。

失败样本才决定“系统能不能进化”。

### 我想解决的核心问题

- 最危险的 false positive 都长什么样？
- 最可惜的 false negative 都集中在哪里？
- 错误是来自执行噪音、regime 失配、信息缺失，还是模型表达能力不足？

### 我建议怎么落地

我会把 casebook 当成正式研究产物，而不是临时调试笔记。

每个失败案例至少记录：

- 事件 ID 与 ticker
- 所属 regime 标签
- `IVAR_event`、模型预测、realized target
- edge 大小与方向
- 执行可信度
- 最终 PnL / hit outcome
- 错误类型标签
- 研究者备注

错误类型标签我会优先从下面几类开始：

- `execution_fragile`
- `regime_mismatch`
- `market_already_right`
- `feature_missing_signal`
- `model_overreaction`
- `model_underreaction`
- `event_contaminated`

### 我希望最终看到什么

- 每轮研究都能沉淀一批失败案例
- 失败案例可以反过来驱动新特征、新过滤器、新分层
- 这个项目不只积累结果，还积累“为什么错”的知识库

这也是我认为它最值钱的原因：

> 它最有机会把这个仓库从“会出结果的研究项目”推进成“会自我修正的研究系统”。

---

## 我对实施顺序的建议

虽然上面是按价值排序，但如果从工程可落地性看，我会这样推进：

### Phase 1：先把研究底座做稳

1. `execution confidence score`
2. `model vs IVAR defeat analysis`
3. `false positive / false negative casebook`

这一阶段的目标不是让结果更漂亮，而是让结果更可信、更可解释、更可复盘。

### Phase 2：再增强结构表达和决策分级

4. `event regime segmentation`
5. `edge confidence calibration`

这一阶段的目标是让系统更清楚地知道“什么地方有效”以及“有效到什么程度”。

---

## 我不建议优先做什么

在这 5 件事之前，我不建议优先继续做下面这些事：

- 再堆更多复杂模型
- 过早追求更复杂的深度网络结构
- 只围绕平均回测结果做优化
- 只看总体指标，不看失败样本和结构分层

原因很简单：

如果执行层、失败归因和市场基准挑战逻辑还没有补强，新增模型大概率只会继续把系统往“更复杂，但不一定更可信”推。

---

## 总结

如果我要用一句话概括我对这套优化路线的判断，那就是：

> 这个项目下一步最应该提升的，不是模型花样，而是结果可信度、结构解释力和失败样本沉淀能力。

我认为真正高价值的升级，不是让系统多输出几个漂亮的回测数字，而是让它更清楚地回答下面这些问题：

- 这些收益到底是不是可成交的？
- 这些 alpha 到底出现在哪些事件机制里？
- 这些模型到底在哪些地方真的打败了市场？
- 这些信号到底有多可靠？
- 这些失败样本到底告诉了我什么？

如果这些问题都能被系统化回答，我会认为这个项目向前迈了一大步。
